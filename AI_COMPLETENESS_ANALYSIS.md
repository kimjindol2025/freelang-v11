## ⚠️ 약점 (개선 필요)

### 1️⃣ 타입 안정성 (68/100) 🔴

**문제**:
```lisp
❌ AI가 생성한 코드
(defun process [data]
  (get data :name))      ;; data가 nil이면? map이면?

❌ 런타임 에러: Cannot read property "name" of undefined
```

**근본 원인**:
- 타입 체크 선택사항 (강제 아님)
- 동적 타입 언어
- 컴파일 타임 검사 없음

**개선 방안**:
```lisp
✅ AI가 생성해야 할 코드
(defun process [data]
  (if (nil? data) (throw "data required"))
  (if (not (map? data)) (throw "data must be map"))
  (let [name (get data :name)]
    (if (nil? name) (throw "name required"))
    name))
```

**필요한 것**:
- [ ] 타입 체크 lint 규칙 추가
- [ ] 에러 발생 시 AI가 자동으로 타입 체크 삽입
- [ ] 선택적 정적 타입 시스템 (미래)

---

### 2️⃣ API 일관성 (75/100) 🟡

**문제 1: 함수 인자 순서 불균형**
```lisp
(map fn list)         ✅ 함수가 먼저
(filter pred list)    ✅ 함수가 먼저
(reduce fn init list) ✅ 함수가 먼저

하지만:
(str-replace string search replace) ❌ 데이터가 먼저
(str-split string delimiter)        ❌ 데이터가 먼저
(get collection key)                ❌ 데이터가 먼저
```

**문제 2: 함수명 표기 불균형**
```lisp
map-entries    ✅ 하이픈
vector-add     ✅ 하이픈
string?        ❌ ?는 suffix (언더스코어로?)
is-number?     ❌ is + ?는 중복
```

**영향**:
- AI가 함수명 추측 어려움
- 인자 순서 혼동 자주 발생

**개선 방안**:
```
우선순위 1: 함수명 일관성
  - 모든 type check는 "string?", "number?" (현재 O)
  - 타입 검증: "is-valid-email", "has-error" (아직 부족)

우선순위 2: 인자 순서 통일
  - 컬렉션 함수: (fn collection ...rest) 통일 검토
  - 문자열 함수: (fn target ...rest) 통일 검토
```
